SignalFoundry case study: security triage and evidence pipeline

SignalFoundry is an AI-augmented security triage and evidence pipeline. The AutoSOC engine is the current Wazuh-based implementation that polls alerts, applies deterministic dispositions, redacts sensitive fields, validates reconciliation and host coverage, and stages analyst-reviewable escalations.

1

Problem

SOC alert queues are dominated by repetitive noise, inconsistent triage decisions, and evidence that is hard to validate after the fact.

2

Goal

Build a repeatable pipeline that can classify recurring alert patterns, preserve suspicious cases, and expose its own operational health.

3

Result

AutoSOC now produces consistent dispositions, redacted outputs, escalation artifacts, and explicit PASS or FAIL state for the pipeline itself.

Architecture
System map Wazuh + AutoSOC

Telemetry enters through Wazuh, moves through AutoSOC triage and validation gates, then exits as either closure history or staged escalation output.

Stack

Core components

  • Proxmox lab with Wazuh, pfSense, endpoints, Grafana, and supporting services.
  • Python and PowerShell automation in the public scripts/auto-soc/ lane.
  • Sigma, Wazuh, and Splunk content tied to the same evidence model.
  • Public proof lane plus site data generated from verified inventory and canonical metrics.
Pipeline logic

How decisions are made

  1. Poll alerts and normalize the case inputs.
  2. Evaluate alert context against deterministic policy rules.
  3. Apply suppression logic for recurring benign or known-false-positive patterns.
  4. Redact sensitive fields before any public-facing artifact is produced.
  5. Validate reconciliation, freshness, heartbeat, and host coverage.
  6. Emit a final disposition and, when needed, stage escalation output for review.
Dispositions

Output classes

Disposition When it is used System action
AUTO_CLOSE_BENIGN Expected activity with sufficient context Close and preserve audit history
AUTO_CLOSE_KNOWN_FP Recurring false-positive pattern already studied Close, annotate, and retain suppression rationale
ESCALATE Suspicious or policy-sensitive case Stage evidence pack and analyst-reviewable escalation artifact

Why the numbers matter

Evidence of tuning, not alert spam

  • High auto-close volume is split between benign activity and known false positives rather than one undifferentiated bucket.
  • Escalations remain preserved as a separate path instead of disappearing inside aggressive suppression.
  • Pipeline health is exposed as explicit gates, so volume numbers are not presented without reliability context.
  • The same site also exposes a separate verified detection inventory of 139 public detections and 10 IR playbooks.
Failure and recovery

Pipeline coverage loss on 03-13-2026

Symptoms: coverage checks failed, alerts stopped arriving, and the pipeline lost confidence in its monitored host set.

Root cause chain: agent connectivity and evidence-pull recovery were followed by a reconciliation-path bug uncovered during retest.

Fix: restore the ingest path, correct the reconciliation-path logic, rerun validation, and confirm the system returned to heartbeat=SUCCESS, reconciliation=PASS, freshness=PASS, and required-host coverage PASS.

Lessons

What this surfaced

  • Security automation needs observability on the pipeline itself, not just the alerts it processes.
  • Reconciliation is a control, not a reporting extra.
  • False-positive tuning only matters if escalation integrity stays intact.
  • Failure handling is part of the engineering signal for SOC automation work.
