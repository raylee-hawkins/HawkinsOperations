Case Study

AutoSOC: alert ingestion to escalation artifacts

Built a SOAR-style Tier-1 triage workflow for Wazuh alerts: policy-based dispositions, mandatory redaction, and incident evidence packs with reviewable GitHub escalation artifacts.

A) Problem

SOC triage work is repetitive and can drift: too many noisy alerts, inconsistent evidence capture, and escalation packets that depend on analyst style instead of a standard output contract.

B) Solution

AutoSOC pipeline flow: Wazuh ingestion -> triage decisioning -> redaction gate -> evidence-pack generation -> escalation via GitHub PR artifacts. The pipeline enforces disposition logic and produces a consistent handoff package for every case.

C) Architecture

Data sources feed ingestion and triage gates, then move through redaction and evidence assembly into reviewable output paths.
Wazuh sourcePolicy gatesEscalation outputs

D) Decision logic

AutoSOC uses explicit dispositions to keep triage behavior deterministic and reviewable.
Disposition Meaning What happens
AUTO_CLOSE_BENIGN Expected activity Close and log evidence summary
AUTO_CLOSE_KNOWN_FP Known false positive pattern Close, annotate reason, keep audit trail
ESCALATE Suspicious/high-confidence case Generate evidence pack and escalation artifacts

E) Example incident

Alert type: high-volume rootcheck alert pattern.
Checks performed: known-FP policy match, repetition pattern, host context, rule history.
Redaction gate: host identifiers and environment-specific values sanitized before publication.
Evidence pack output: one-pager, full report, timeline CSV, queries file, closure report.
Disposition: repeat benign pattern moved to AUTO_CLOSE_KNOWN_FP after validation and tuning. 
rootcheck_window_alerts: 1312
dominant_pattern: repeated benign rootcheck signatures
action: policy tuning + replay validation
outcome: reduced false-positive pressure while preserving escalation path
