Pipeline Architecture

How AutoSOC works

Step 1 — Alert Intake

poll-alerts.py queries the Wazuh indexer REST API on a schedule and writes new alerts to a local queue file. Each record includes rule ID, severity level, description, and agent metadata consumed by the triage step.

Step 2 — Triage Decision

triage.py evaluates each queued alert against policy.yaml and known_fps.yaml. Output is one of three deterministic dispositions: AUTO_CLOSE_BENIGN, AUTO_CLOSE_KNOWN_FP, or ESCALATED. No manual input required per alert.

Step 3 — Redaction + Evidence

redact.py strips real IPs, hostnames, and identifiers before data leaves the private environment. assemble-pack.py generates six report documents from the sanitized data: timeline, triage summary, IOC list, remediation steps, MITRE mapping, and closure report.

Step 4 — Escalation Output

create-pr.py stages the complete incident pack and opens a GitHub PR. Escalated cases are publicly reviewable with full redacted artifacts. run-pipeline.py orchestrates the full sequence on a Task Scheduler trigger.
